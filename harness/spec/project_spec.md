# ChatBI 项目规范

## 项目名称

**ChatBI** — 基于自然语言交互的智能数据分析与报告生成系统

---

## 项目背景

在各职能与业务领域（供应链、财务、销售、人力资源等），日常分析与决策都依赖多系统、多口径的数据报表。业务人员的典型工作流程是：

1. 从不同系统（ERP、CRM、WMS、POS、财务系统等）或既有报表中反复查找、筛选与拼接数据维度
2. 按具体需求进行二次加工——Excel 公式、数据透视、跨表关联、手工计算指标
3. 最终生成定制化报表、指标看板（Dashboard）或分析报告（PDF）

### 核心痛点

| 痛点 | 具体表现 |
|------|----------|
| **耗时耗力** | 每次新需求都从零开始找数据、对口径、写公式，大量重复劳动 |
| **易出错** | 手工操作导致公式错误、引用遗漏、版本混乱，数据质量难以保证 |
| **口径难统一** | 不同人员、不同时间对同一指标的计算方式不一致，结果不可比 |
| **数据响应慢** | 从"提出需求"到"拿到报表"往往需要数小时甚至数天 |
| **复用率低** | 每次分析产出难以沉淀为可复用的模板、流程或知识 |
| **难以支撑快速迭代** | 管理层与业务需求变化快，手工模式响应速度跟不上 |

### 目标用户

- 业务分析师：需要快速从数据中获取洞察
- 部门管理者：需要定期查看经营指标看板
- 数据团队：需要将重复性报表工作自动化
- 决策层：需要及时、准确的数据支撑决策

---

## 项目目标

构建 **ChatBI 智能分析平台**，实现以下核心能力：

### 1. 自然语言 → 数据查询（NL2SQL / NL2Data）

用户以自然语言描述分析需求，系统自动完成：
- 意图识别：理解用户想分析什么
- 表与字段匹配：从多源数据中找到正确的数据表与字段
- 聚合逻辑生成：自动推断聚合方式（求和、计数、平均值、同比、环比等）
- 多口径兼容：当同一指标存在多种计算口径时，明确告知并让用户选择

### 2. 自动数据提取与处理

- 从多源数据（数据库、CSV、Excel、API）中自动提取所需数据
- 自动处理数据清洗、关联、转换
- 对数据质量进行内置校验

### 3. 自动生成分析产物

| 产物类型 | 说明 |
|----------|------|
| **Dashboard** | 交互式指标看板，支持钻取、筛选、联动 |
| **PDF 分析报告** | 格式化专业报告，含图表、表格、洞察文字 |
| **Excel/CSV 数据文件** | 结构化数据提取，供下游二次使用 |

### 4. 全链路质量保证

通过 Harness Engineering 六层架构，确保：
- 每次分析结果可追溯至源数据
- 每个阶段的输出通过质量门校验
- 相同问题得到一致的结果
- 错误可被捕获、记录并防止复发

## 成功标准

| 维度 | 标准 | 量化指标 |
|------|------|----------|
| **准确性** | 生成的数据查询正确 | NL2SQL/聚合逻辑准确率 ≥ 95% |
| **效率** | 大幅缩短报表产出时间 | 目标：从"小时/天"级降至"分钟"级 |
| **一致性** | 相同问题得到相同结果 | 口径可追溯，同口径偏差 < 1% |
| **覆盖度** | 支持多业务领域 | 框架可接入 ≥ 3 个业务领域数据源 |
| **可用性** | 非技术人员可独立使用 | 自然语言输入即可完成分析，无需编写SQL或代码 |
| **可靠性** | 系统稳定输出 | 全流程成功率 ≥ 98% |

## 项目范围

### 包含（In Scope）

- 自然语言解析与数据分析意图识别
- 多源数据连接与查询自动生成
- 数据可视化与 Dashboard 自动排版
- PDF 分析报告自动生成
- Harness Engineering 全链路质量保证框架
- 案例验证（以零售库存数据为首个验证案例）

### 暂不包含（Out of Scope — 本期）

- 实时流数据处理（本期仅支持批处理数据）
- 对话式自然语言回复（仅生成数据产物，不做多轮对话答疑）
- 数据治理与元数据管理平台
- 移动端原生应用
- 多租户SaaS化部署

## 框架设计原则：不变与可变

Harness Engineering 框架区分两个层次：

### 不变的骨架（六层目录结构）

以下 L1-L6 六层目录结构是**永久固定的**，不因业务主题变化：

```
harness/
├── spec/          # L1 信息边界层 — 项目/数据/任务规范
├── skills/        # L2 工具系统层 — 标准化可复用流程
├── agents/        # L3 执行编排层 — 子Agent角色定义
├── workflows/     # L3 执行编排层 — 工作流定义
├── memory/        # L4 记忆与状态层 — 决策记录/经验教训
├── evaluation/    # L5 评估与观测层 — 质量门/评估指标
├── rules/         # L6 约束层 — 硬编码红线
└── scripts/       # L6 校验层 — 不可绕过的硬校验
```

这八项目的骨架为每一次分析提供统一的工程框架参考，建议保持完整——即使当前任务较简单导致部分目录暂时留空，结构本身也值得保留。

### 可变的内容（随业务场景调整）

以下内容**建议**根据每次具体的业务数据和任务目标进行调整：

| 可变项 | 说明 | 示例（库存案例） |
|--------|------|-----------------|
| **Agents（子Agent角色）** | 角色种类与数量由业务分析所需的专业分工决定 | 数据工程师、预测分析师、库存分析师、定价分析师、审查员 |
| **Workflows（工作流）** | 工作流数量与阶段划分由分析链路的复杂度决定 | data_prep → forecast → inventory → pricing |
| **Phases（分析阶段）** | 阶段数量与内容由业务问题的分解方式决定 | EDA → 预测审计 → ABC/XYZ → 安全库存 → 促销分析 → 竞争定价 → Dashboard |
| **Skills（标准流程）** | 技能文件的内容由该业务领域的具体分析方法决定 | 预测评估流程、ABC分类标准、可视化规范 |
| **Metrics（评估指标）** | 指标集由业务领域的行业标准和目标决定 | MAPE/WMAPE/Service Level/Turnover/ROI |
| **Quality Gates（质量门）** | `case:` 部分的检查项由业务案例的验收标准决定 | 预测优于基线、SL达标、弹性显著、ROI>1 |
| **Task Specs（任务规范）** | `harness/spec/tasks/` 中的文件内容随业务主题调整 | 7个库存分析任务规范 |

> 直接复用既有配置也可开始分析，建议至少审查一遍是否适配当前任务。每次启动时 `check_task_readiness.py` 会自动扫描并给出建议。

### 新案例接入流程

当 ChatBI 接入新的业务案例时，按以下步骤操作：

| 步骤 | 操作 | 涉及文件 |
|------|------|----------|
| 1. 注册案例 | 在本文档"案例数据"表中追加新案例条目 | `project_spec.md` |
| 2. 数据规范 | 按模板追加新案例的数据字典和质量基线 | `data_spec.md` |
| 3. 案例 Agent | 创建或修订案例专用 Agent 定义 | `harness/agents/*.md` |
| 4. 案例 Workflow | 创建案例专用的工作流 YAML | `harness/workflows/*.yaml` |
| 5. 任务规范 | 根据业务问题分解，创建新的任务 spec 文件 | `harness/spec/tasks/*.md` |
| 6. 案例指标 | 创建案例专用的评估指标文件 | `harness/evaluation/metrics/*.py` |
| 7. 质量门 | 在 `case:` 下追加新案例的检查项 | `harness/evaluation/quality_gates.yaml` |
| 8. 校验脚本 | 如需自动化校验，创建案例专用的检查脚本 | `harness/scripts/check_*.py` |

> ⚠️ 平台层文件（🔵 标记）不随案例变化，案例层文件（🟠 标记）需按此流程逐一更新。

### 执行建议

> **每次新任务开始时，建议先阅读本 `project_spec.md` 了解项目背景，然后运行 `check_task_readiness.py` 扫描当前项目结构。扫描结果中的可改进项不会阻塞分析继续，但可作为补充完善的参考。**

### L4 记忆层自动更新要求

L4 层（`harness/memory/`）是 Harness 的持续记忆系统，要求**在每次任务执行中自动更新**：

- **`state/current.md`**：每个 Workflow 的 stage 执行前后更新进度状态和 checklist
- **`decisions.md`**：每次技术选型、架构调整、参数定版时追加决策记录
- **`lessons.md`**：每次遇到错误、reviewer 驳回、基础设施问题时追加经验教训

> 原则：记忆层不是"事后总结"，而是"执行中持续沉淀"。棘轮只进不退。

## 关键约束

1. **数据安全**：用户仅能查询其权限范围内的数据表与字段，不可越权访问
2. **口径统一**：所有指标定义必须在当前案例的 `data_spec.md` 中注册，不可临时编造口径
3. **质量门不可绕过**：每次分析必须通过 `quality_gates.yaml` 中定义的全部检查项
4. **六层目录建议保持**：无论分析何种业务数据，建议保持 L1-L6 八项目录结构，当前任务较简单时部分目录可暂时留空
5. **Agents/Workflows/Phases 建议按需调整**：每次新任务建议根据具体业务数据和目标审查这三个可变维度，直接复用既有配置也可开始
6. **案例驱动开发**：以具体业务案例验证框架有效性，首个案例为零售库存分析
7. **建议不阻塞**：`check_task_readiness.py` 在每次 workflow 启动前扫描并给出建议，始终允许继续执行

## 案例数据

为验证 ChatBI 框架的完整性和有效性，本项目使用以下业务案例进行开发与测试：

| 案例 | 数据集 | 规模 | 业务领域 |
|------|--------|------|----------|
| **库存分析** | Retail Store Inventory Forecasting (Kaggle) | 73,100 行 × 15 列 | 零售供应链 |

> 后续可扩展更多业务领域案例（如销售分析、财务分析、客户分析等），每个案例均在相同的 Harness 框架下执行。
> 无论分析何种数据，框架六层架构保持不变，仅 `data_spec.md` 和 `tasks/` 中的具体内容随业务主题调整。
